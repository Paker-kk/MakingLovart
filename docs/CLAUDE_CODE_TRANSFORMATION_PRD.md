# Flovart Claude Code 改造 PRD

> 版本：v0.1
>
> 日期：2026-04-10
>
> 文档目标：把 Flovart 从“带 AI 生成能力的创作工具”推进为“可由 Claude Code 编排的 AI 短剧样片生产系统”

---

## 一、产品一句话

输入一句话创意或一篇长文本，自动生成可观看的短剧样片；所有中间结果与最终结果都回到 Flovart 画布，允许用户随时暂停、修改、补拍、继续生产。

---

## 二、为什么要做这次改造

### 2.1 当前 Flovart 的强项

| 能力层 | 当前已有能力 | 价值 |
|------|------|------|
| 多模型生成 | 文生图、图生图、视频生成、Provider 路由 | 已具备生产基础设施 |
| 工作流编排 | Node Workflow、RunningHub、模板执行 | 已具备可扩展工作流能力 |
| 画布与资产管理 | 画布编辑、图层、素材库、面板系统 | 已具备人工接管基础 |
| 多代理讨论 | `agentOrchestrator.ts` 已支持多角色讨论 | 已具备协作式 prompt 生成基础 |

### 2.2 当前 Flovart 的短板

| 问题 | 现状 | 后果 |
|------|------|------|
| 缺少短剧级任务入口 | 现在仍以图像生成与通用创作为主 | 用户无法一眼理解这是短剧生产系统 |
| 缺少稳定批量出片机制 | 有工作流基础，但没有围绕“样片生产”打磨 | 结果可用性不稳定 |
| 缺少面向 Claude Code 的能力封装 | 当前能力多分散在 UI 与 service 中 | 难以被外部代理稳定编排 |
| 缺少“暂停-修改-继续”闭环 | 用户能手动改，但没有流程级暂停点和恢复点 | 可控性不足 |

### 2.3 为什么参考 MeiGen，但不能照抄 MeiGen

MeiGen 的优势不在于“它有 MCP”，而在于以下产品组合：

1. 插件化分发路径短，安装门槛低。
2. 命令化工作流明确，用户容易理解。
3. 子代理职责清晰，天然适合并行任务。
4. 后端可替换，支持本地与云端生成。
5. 免费能力先体验，付费能力后转化。

Flovart 的目标不同。MeiGen 是“设计图生成增强层”，Flovart 要做的是“短剧样片生产系统”。

因此，Flovart 必须把重心放在：

1. 剧情拆解与分镜规划。
2. 角色一致性与镜头连续性。
3. 批量稳定出片。
4. 画布内导演级可编辑闭环。

## 三、产品定位

### 3.1 定位陈述

Flovart Claude Code 改造版，不是通用图像生成插件，也不是 Lovart 的简单替代品，而是：

**一个可以由 Claude Code 驱动、面向 AI 短剧样片生产的可编辑工作台。**

### 3.2 北极星能力

用户输入一句话创意或一篇长文后，系统完成：

1. 剧情结构拆解。
2. 分镜与镜头清单规划。
3. 角色、风格、节奏约束生成。
4. 图像与视频批量生产。
5. 结果自动回写到 Flovart 画布。
6. 用户在关键节点暂停修改后继续执行。

### 3.3 目标用户

1. AI 短剧创作者。
2. 小型内容工作室与制片团队。
3. 需要快速验证故事与镜头方案的品牌内容团队。

### 3.4 非目标用户

1. 只想做单张图片的轻量创作者。
2. 只需要聊天式 prompt 优化而不需要生产闭环的用户。
3. 首期就要求商用终片质量的影视工业客户。

---

## 四、核心价值主张

### 4.1 对用户的直接价值

1. 把“创意到样片”的时间缩短到可演示、可迭代的周期。
2. 不必在多个工具之间手工搬运剧情、分镜、图像和视频。
3. 可以像导演一样在画布里接管流程，而不是被黑盒一键生成绑架。

### 4.2 相对竞品的差异化

| 维度 | MeiGen | Flovart Claude Code 改造版 |
|------|------|------|
| 核心对象 | 设计图 / 创意图工作流 | 短剧样片生产工作流 |
| 结果类型 | 图片方向探索 | 剧情、分镜、图像、视频、样片 |
| 可控性 | 主要在 prompt 与工具编排层 | 画布内导演级修改与节奏控制 |
| 护城河 | 设计工作流与提示词资产 | 短剧语法引擎 + 画布闭环 + 稳定批量出片 |

## 五、MVP 范围

### 5.1 首期必须打通的能力

1. 一句话或长文生成短剧样片。
2. 固定工作流下的稳定批量出片。
3. 关键节点暂停、修改、继续执行。
4. 单镜头重生成与补拍。
5. 所有结果落到 Flovart 画布。

### 5.2 首期必须保留的人工干预点

1. 角色设定干预。
2. 分镜清单干预。
3. 提示词干预。
4. 重生成干预。
5. 成片节奏干预。

### 5.3 首期不做

1. 重度桌面自动点击控制。
2. 多人协作与审批系统。
3. 商业级终片质量承诺。
4. 面向全部创意领域的通用化插件矩阵。

## 六、核心用户流程

### 6.1 主流程

1. 用户输入一句话创意，或导入一篇长文。
2. Claude Code 调用短剧规划能力，生成剧情结构与节奏方案。
3. 系统生成分镜清单、镜头说明、风格与角色约束。
4. 用户在 Flovart 中快速确认或局部修改。
5. 系统按工作流批量生成分镜图与关键视频镜头。
6. 所有素材自动进入画布与资产区。
7. 用户选择继续执行、单镜头补拍、修改节奏或暂停。
8. 系统继续产出样片版本。

### 6.2 失败恢复流程

1. 某镜头生成失败。
2. 系统标记失败原因：超时、风格漂移、角色不一致、视频不可用等。
3. 用户选择自动重试、手动改 prompt、切换模型或跳过。
4. 系统只重跑失败镜头，不重跑整条流程。

## 七、产品模块设计

### 7.1 剧情拆解模块

输入：一句话创意或长文故事。

输出：

1. 剧情段落。
2. 角色关系。
3. 冲突曲线。
4. 节奏结构。
5. 每段剧情的镜头目标。

### 7.2 分镜规划模块

输出：

1. 镜头顺序。
2. 景别与机位。
3. 时长与节奏建议。
4. 情绪与叙事功能。
5. 每镜头图像与视频的生成需求。

### 7.3 视觉约束模块

输出：

1. 角色视觉设定。
2. 服化道连续性。
3. 场景一致性。
4. 风格统一约束。

### 7.4 生产执行模块

职责：

1. 调用图像生成。
2. 调用视频生成。
3. 调用工作流模板。
4. 批量并行处理。
5. 失败重试与回退。

### 7.5 审片与补拍模块

职责：

1. 检测缺镜头。
2. 检测角色漂移。
3. 检测风格不一致。
4. 检测节奏异常。
5. 输出补拍建议。

### 7.6 画布回写模块

职责：

1. 把每个镜头结果放到 Flovart 画布。
2. 保留与分镜节点的映射关系。
3. 允许暂停后继续接管执行。

## 八、Claude Code 改造方案

### 8.1 改造目标

让 Claude Code 不只是“在仓库里写代码”，而是能够以稳定、可分发、可组合的方式调用 Flovart 的短剧生产能力。

### 8.2 首期交付形态

首期建议采用：

**项目内插件骨架 + 后续可平滑迁移到分发型插件/marketplace**

原因：

1. 当前验证速度优先。
2. 需要先跑通能力边界与执行入口。
3. 后续再升级为真正可分发的 Claude Code plugin。

### 8.3 Claude Code 侧能力构成

1. `skills`：封装短剧任务流。
2. `agents`：拆分剧情、分镜、视觉、执行、审片角色。
3. `hooks`：对高风险动作、自动校验、自动归档做约束。
4. `commands`：提供 setup、run、storyboard、review 等用户入口。
5. `mcpServers`：第二阶段接入，用于标准化外部工具调用。

### 8.4 推荐的首批命令

1. `setup`：配置模型、Provider、默认节奏、输出目录。
2. `run-shortdrama`：一句话或长文生成短剧样片。
3. `storyboard`：只生成剧情与分镜包。
4. `batch-produce`：对多个镜头并行出片。
5. `review-dailies`：审片并生成补拍任务。

### 8.5 推荐的首批代理

1. `story-architect`：剧情结构拆解。
2. `storyboard-director`：分镜规划。
3. `visual-supervisor`：角色与风格一致性。
4. `production-runner`：图像/视频/工作流执行。
5. `dailies-reviewer`：审片与补拍建议。

## 九、代码改造方案

### 9.1 改造原则

1. 优先复用现有 service 和 workflow 能力。
2. 不让自动化逻辑直接绑死在 UI 行为上。
3. 先做稳定执行入口，再做更复杂的前端表现。
4. 所有结果回写画布，而不是只在日志或对话中返回。

### 9.2 现有代码的主要改造落点

#### A. `services/workflowEngine.ts`

改造目标：

1. 增加短剧流水线执行包装层。
2. 支持“剧情拆解 → 分镜 → 图像 → 视频 → 回写”的标准化执行。
3. 支持批量任务状态、失败重试、结果索引。

#### B. `services/agentOrchestrator.ts`

改造目标：

1. 从“图像 prompt 讨论器”升级为“短剧多代理编排器”。
2. 支持剧情、分镜、视觉、审片多角色协作。
3. 支持按阶段输出结构化结果，而不是只输出最终 prompt。

#### C. `components/NodeWorkflowPanel.tsx`

改造目标：

1. 把现有工作流模板能力转换成短剧任务模板入口。
2. 支持从“样片任务”自动实例化工作流。
3. 展示批量执行进度与阶段状态。

#### D. `components/RightPanel.tsx`

改造目标：

1. 增加短剧任务队列视图。
2. 增加失败镜头补拍入口。
3. 增加暂停/继续执行控制。
4. 增加角色、分镜、节奏编辑面板入口。

#### E. `App.tsx`

改造目标：

1. 接入短剧任务态。
2. 管理样片工作流与画布回写状态。
3. 统一协调侧边栏、任务队列、资产回写、生成状态。

### 9.3 新增目录建议

建议新增：

```text
.claude/
  agents/
  skills/
  settings.json

claude-plugin/
  .claude-plugin/
    plugin.json
  agents/
```

说明：

2. `claude-plugin/` 用于后续可分发版本。
3. `services/shortDrama*.ts` 用于沉淀可被 UI、插件、工作流三方共用的能力。

## 十、MVP 分阶段实施建议

### Phase 1：骨架期

目标：先把“短剧任务”从概念变成可调用的系统入口。

交付：

1. 短剧 pipeline service 骨架。
2. Claude Code skills / agents 初版。
3. 最小命令入口。
4. 短剧任务状态数据结构。

### Phase 2：跑通期

目标：打通一句话到样片的最小链路。

交付：

1. 剧情拆解。
2. 分镜清单生成。
3. 批量生成静态分镜图。
4. 回写 Flovart 画布。

### Phase 3：稳定期

目标：解决你当前最关心的“质量优先 + 稳定出片”。

交付：

1. 固定工作流模板。
2. 失败重试策略。
3. 单镜头补拍。
4. 角色一致性约束。

### Phase 4：增强期

目标：把静态分镜能力推进到视频样片能力。

交付：

1. 首尾帧视频生成。
2. 节奏控制。
3. 审片与补拍建议。
4. 插件化分发准备。

## 十一、成功指标

### 11.1 核心指标

1. 固定工作流下的稳定出片率。
2. 批量任务吞吐量。
3. 单镜头补拍替代整链路重跑的比例。
4. 样片首轮可观看率。
5. 用户暂停后继续执行的成功率。

### 11.2 用户体验指标

1. 用户是否理解这是短剧生产系统，而不是图片工具。
2. 用户是否能在画布中完成主要人工修正。
3. 用户是否能快速找到失败镜头并单独处理。

---

## 十二、主要风险

### 风险 1：定位滑回“通用 AI 创作工具”

后果：产品叙事发散，无法建立清晰心智。

应对：所有新功能先问一句，它是否直接提升短剧样片生产能力。

### 风险 2：自动化路径过度依赖 UI 操作

后果：流程脆弱，难以稳定运行。

应对：优先构建 service 级执行入口，再让 UI 消费这些结果。

### 风险 3：批量生成质量不稳定

后果：吞吐高但不可用。

应对：优先做固定工作流模板、结构化输入、失败补拍。

### 风险 4：结果不能在画布内高效接管

后果：Flovart 的可编辑优势无法体现。

应对：把回写与暂停继续能力列为 MVP 必做项。

---

## 十三、当前必须继续追问的关键问题

1. 首期“样片可看”的最低质量标准是什么。
2. 角色一致性与节奏质量，哪个优先级更高。
3. 用户接受的默认生成耗时上限是多少。
4. 用户在 Flovart 中最常修改的是角色、镜头、节奏还是 prompt。
5. 未来要不要做成真正可分发的 marketplace plugin，而不是仓库内能力包。

---

## 十四、结论

这次改造的本质，不是把 Claude Code 接到 Flovart 上，而是把 Flovart 升级成一个可被 Claude Code 编排的短剧样片生产系统。

真正的护城河不在于：

1. 有没有 MCP。
2. 能不能自动点界面。
3. 能不能多接几个模型。

真正的护城河在于：

1. 短剧语法拆解与分镜规划。
2. 稳定的批量出片能力。
3. 画布内的人工可编辑闭环。
4. 失败可恢复、流程可继续的生产级体验。
